Why use Icon? #

You might choose Icon for several reasons:

  • Powerful Pattern Matching: Icon’s pattern matching capabilities significantly simplify complex data manipulation tasks, making code more readable and maintainable. This is particularly beneficial when dealing with strings, lists, and trees.
  • Elegant Exception Handling: Icon’s approach to exception handling is more integrated and less disruptive than in many other languages. It utilizes the concept of “failure” as a natural part of program flow, leading to more concise and efficient code.
  • High-Level Data Structures: Icon provides built-in support for sophisticated data structures, reducing the need for manual management and making complex data organization easier.
  • Goal-Oriented Programming: Icon’s design encourages a goal-oriented programming style, focusing on what needs to be accomplished rather than how to achieve it. This can lead to more declarative and easier-to-understand code.
  • Strong String Handling: Icon offers extensive built-in functions for string manipulation, making it ideal for text processing applications.

Writing your first Icon smart contract #

This section guides you through creating and deploying a simple Icon smart contract. Icon uses a language called SCORE (Smart Contract On Reliable Environment) for writing smart contracts.

Understanding Icon’s smart contract language (SCORE) #

SCORE is a purpose-built language for developing smart contracts on the Icon blockchain. It’s designed for security and reliability, incorporating features that help prevent common vulnerabilities. SCORE is based on Python, making it relatively easy to learn for developers familiar with Python. However, it has some key differences and limitations to ensure the security and integrity of the Icon network. Key aspects include:

  • Type Safety: SCORE emphasizes type safety, helping to prevent unexpected errors and vulnerabilities.
  • Limited External Interactions: SCORE restricts access to external resources to mitigate security risks.
  • Deterministic Execution: SCORE contracts must always produce the same output for the same input, ensuring consistent and predictable behavior.
  • Gas Mechanism: Similar to other blockchains, SCORE uses a gas mechanism to manage the computational resources used by contracts. This prevents contracts from consuming excessive resources and ensures fair network usage.

This guide focuses on a basic understanding; deeper dives into specific features and advanced techniques are available in the complete Icon SCORE documentation.

Basic SCORE structure #

A basic SCORE consists of a Python file containing a set of functions that define the contract’s functionality. The file typically follows a specific structure, including:

  • __init__.py: This file is essential and usually contains the initialization logic for the contract, setting up the initial state.
  • Other Python files: These files hold additional functions and classes used by the contract.
  • icon_score_main.py: This is the main entry point for the contract. It defines core functions like on_install, on_update, and other external functions accessible by users.

Let’s look at a minimal example:

# icon_score_main.py
from iconservice import *

class HelloWorld(IconScoreBase):
    _hello = 'Hello, world!'

    def __init__(self, db: IconScoreDatabase) -> None:
        super().__init__(db, db)
        self._hello = VarDB('hello', db, value_type=str)

    def on_install(self) -> None:
        super().on_install()

    @external
    def get_message(self) -> str:
        return self._hello.get()

    @external
    def set_message(self, new_message: str) -> None:
        self._hello.set(new_message)

This contract has two external functions: get_message retrieves a stored message and set_message updates it. Notice the use of @external decorator, indicating functions callable from outside the contract.

To deploy your contract, you will need:

  1. Compile the SCORE: Use the Icon SDK’s compiler to compile your SCORE code into a deployable package (usually a .zip file).
  2. Prepare the Deployment Transaction: Use the Icon SDK to create a transaction to deploy your contract. This will involve specifying the contract’s bytecode, initial parameters, and other necessary details.
  3. Send the Deployment Transaction: Submit the deployment transaction to the Icon network using your wallet.

The exact commands and steps for compilation and deployment will depend on the Icon SDK version and your chosen development tools. Refer to the official Icon documentation for specific instructions.

Interacting with your deployed contract #

After deploying the contract, you can interact with its functions using the Icon SDK. This involves:

  1. Obtain the Contract Address: The deployment transaction will return the address of your newly deployed contract.
  2. Call Contract Functions: Use the SDK’s methods to call the external functions of your contract, such as get_message and set_message in our example.
  3. Send Transactions: If a function modifies the contract’s state (e.g., set_message), you’ll need to send a transaction to execute it. This requires signing the transaction with your wallet’s private key.

Once again, the precise methods for interacting with your contract will depend on your chosen SDK and tools. Consult the relevant documentation for details.

Understanding the Icon Network #

This section provides a high-level overview of the Icon network’s architecture and key components.

Network architecture #

The Icon network employs a unique architecture designed for scalability and interoperability. It’s a hybrid architecture combining features of both public and private blockchains. Key elements include:

  • Relay Chains: These chains handle the consensus mechanism and maintain the overall network integrity. They are responsible for validating and confirming transactions across the network.
  • Community Chains (or “Branch Chains”): These are independent blockchains that can be created and managed by individual organizations or communities. They can benefit from the security and stability of the relay chain, while enjoying a degree of customization and autonomy. They are connected to the main relay chain to allow interoperability.
  • Interoperability: Icon’s architecture prioritizes interoperability, allowing various community chains to communicate and interact with each other in a controlled manner. This facilitates the creation of decentralized applications (dApps) that can leverage the resources and functionalities of different chains.

Consensus mechanism (Loopchain) #

Icon utilizes the Loopchain consensus mechanism, a variation of Practical Byzantine Fault Tolerance (PBFT). Loopchain is designed to achieve high throughput and low latency while maintaining a high level of security. It differs from traditional PBFT in its ability to handle a larger number of nodes efficiently. Key characteristics of Loopchain include:

  • High Throughput: Loopchain aims to achieve a higher transaction processing rate compared to some other consensus mechanisms like Proof-of-Work (PoW) or even some variations of Proof-of-Stake (PoS).
  • Low Latency: Transactions are confirmed relatively quickly, leading to a better user experience.
  • Fault Tolerance: The system is designed to be resilient to node failures, ensuring the network continues to operate even if some nodes malfunction.
  • Deterministic State Transition: The order of transactions and state transitions is always consistent and predictable, which is crucial for smart contract execution.

Block structure and transactions #

An Icon block contains a collection of validated transactions. The structure is similar to other blockchains but may have some specific Icon-related fields. A typical block includes:

  • Block Header: Metadata such as block height, timestamp, hash of the previous block, and other relevant information.
  • Transactions: A list of transactions included in the block. Each transaction includes details such as sender, receiver, amount of ICX (if applicable), and data for smart contract calls.

Transactions in Icon typically follow a standard format, specifying the sender, receiver, amount of ICX (if any), and any data related to a smart contract interaction. They are verified by the consensus mechanism and included in a block once validated.

ICX tokens and transaction fees #

ICX is the native cryptocurrency of the Icon network. It’s used for:

  • Transaction Fees: Users pay transaction fees in ICX to incentivize network participants (validators) to process their transactions. The fees are relatively low.
  • Staking: ICX holders can stake their tokens to become validators and participate in the consensus mechanism. They earn rewards in ICX for their participation.
  • Smart Contract Interactions: Some smart contracts may require users to pay fees in ICX to interact with their functionalities.

The transaction fees depend on several factors, including the complexity of the transaction and the network congestion. The amount is usually small and designed to be accessible to all users. Stake rewards are an essential part of the network’s security and incentivize users to participate in maintaining the blockchain.

Working with data on Icon #

This section details how to manage and interact with data within your Icon SCORE (Smart Contract On Reliable Environment) projects.

Data types in SCORE #

SCORE supports a range of data types, largely mirroring those found in Python. Understanding these types is crucial for writing efficient and type-safe contracts. Key data types include:

  • Integers (int): Represent whole numbers. Note that there might be limitations on the maximum and minimum values depending on the implementation.
  • Strings (str): Represent text. Strings are immutable; modifying a string creates a new string object.
  • Booleans (bool): Represent truth values (True or False).
  • Arrays (list): Ordered collections of elements. Elements can be of different types, but type consistency within a single array is generally recommended for better code readability and maintainability.
  • Dictionaries (dict): Unordered collections of key-value pairs. Keys must be immutable (e.g., strings, numbers), while values can be of any type.
  • Bytes (bytes): Represent raw binary data. This is important when working with on-chain data that may not be easily represented using standard text or numerical formats.
  • Address (Address): A special type representing an Icon address. It is used to identify accounts and contracts on the Icon network.

Storing and retrieving data #

SCORE provides mechanisms for storing and retrieving data persistently within the contract’s storage. This is essential for maintaining state and managing contract data across multiple transactions. The primary method is using the IconScoreDatabase. Variables are typically declared using VarDB (for single values) or other database structures provided by the Icon SDK.

Example:

from iconservice import *

class MyContract(IconScoreBase):
    _my_variable = VarDB('my_var', db, value_type=int)

    def __init__(self, db: IconScoreDatabase) -> None:
        super().__init__(db, db)

    @external
    def set_data(self, value: int) -> None:
        self._my_variable.set(value)

    @external
    def get_data(self) -> int:
        return self._my_variable.get()

In this example, _my_variable stores an integer value. set_data updates the value, and get_data retrieves it.

Using databases (e.g., LevelDB) #

Icon SCORE contracts typically leverage LevelDB, an embedded key-value store, for persistent data storage. While the VarDB example above abstracts away the LevelDB interaction, understanding LevelDB’s properties is beneficial for efficient data management.

LevelDB allows you to store data as key-value pairs. Keys must be unique, and choosing appropriate keys is essential for efficient retrieval. The underlying LevelDB is managed by the IconScoreDatabase. Direct interaction with LevelDB is generally discouraged except for very specific cases requiring advanced data structures or very fine-grained control. The Icon SDK provides higher-level abstractions (like VarDB, DictDB, ArrayDB) that should be preferred for most scenarios.

Event logging #

Event logging is a crucial mechanism for informing external systems and applications about changes in the contract’s state. Events are emitted using the Logger class and can be indexed and queried using tools like Icon Explorer.

Example:

from iconservice import *

class MyContract(IconScoreBase):
    # ... other code ...

    @external
    def my_function(self, value: int):
        # ... some logic ...
        self.MyEvent(value) # Emit event

    @eventlog
    def MyEvent(self, my_value: int):
        pass # Event declaration.  Nothing needs to go here for simple events.

In this example, MyEvent is declared as an event using @eventlog. When my_function is called, self.MyEvent(value) emits an event containing the provided value. External applications or tools can subscribe to these events and react accordingly. You can include multiple parameters in the MyEvent declaration and include them in the event call. Remember to follow the correct naming conventions and types defined in the Icon documentation.

Advanced Concepts #

This section covers more advanced topics related to Icon SCORE development, focusing on security, testing, best practices, and troubleshooting.

Security considerations #

Security is paramount when developing smart contracts. A compromised contract can lead to significant financial and reputational damage. Key security considerations include:

  • Input Validation: Always validate all inputs received from external calls. Never trust user-supplied data without thorough validation. This includes checking data types, ranges, and lengths to prevent unexpected behavior or exploits.
  • Access Control: Implement appropriate access control mechanisms to restrict access to sensitive functions or data. Use modifiers or dedicated functions to control who can interact with specific parts of your contract.
  • Reentrancy Attacks: Be aware of reentrancy vulnerabilities. Reentrancy occurs when a malicious contract calls back into your contract before your contract has completed its execution, potentially manipulating its state. Use techniques like the “Checks-Effects-Interactions” pattern to mitigate this risk.
  • Arithmetic Overflow/Underflow: Be cautious when performing arithmetic operations, particularly with large numbers. Use libraries or safe arithmetic functions to avoid potential overflow or underflow errors.
  • Denial-of-Service (DoS) Attacks: Design your contract to be resistant to denial-of-service attacks. This may involve carefully managing resource consumption and avoiding overly complex or computationally expensive operations.
  • Use of Trusted Libraries: Only utilize well-vetted and audited libraries from trusted sources. Thoroughly review the code of any external library before incorporating it into your contract.
  • Regular Audits: Consider professional security audits of your contract code to identify potential vulnerabilities before deployment.

Testing and debugging SCOREs #

Thorough testing is crucial to ensure the correctness and security of your SCORE. Testing methodologies include:

  • Unit Tests: Test individual functions in isolation to verify their behavior. Unit tests should be comprehensive and cover all possible scenarios, including edge cases and error conditions.
  • Integration Tests: Test the interaction between different functions within your contract. Integration tests ensure that different parts of your contract work together correctly.
  • End-to-End Tests: Test the entire contract flow, including deployment and interaction from external clients. End-to-end tests simulate real-world usage scenarios.
  • Debugging Tools: Use debugging tools provided by your IDE or the Icon SDK to identify and resolve errors in your code. Step-through debugging can help you understand the execution flow and identify the root cause of problems.

The Icon SDK usually provides tools and frameworks to assist with testing. Refer to the SDK documentation for specific instructions.

Best practices for contract development #

Following best practices improves code quality, maintainability, and security:

  • Modular Design: Design your contracts in a modular way, separating concerns into smaller, reusable components.
  • Code Comments: Write clear and concise comments to explain your code’s logic and intent. This improves code readability and makes it easier for others to understand and maintain your contract.
  • Version Control: Use a version control system (e.g., Git) to manage your code and track changes.
  • Code Reviews: Have other developers review your code to identify potential issues and improve code quality.
  • Clear Naming Conventions: Use consistent and meaningful names for variables, functions, and other code elements. Follow the Icon SDK’s naming conventions whenever possible.
  • Error Handling: Implement robust error handling to gracefully handle unexpected situations and prevent crashes.
  • Upgradability: Consider how you can upgrade your contract in the future to add new features or fix bugs without requiring a full redeployment.

Troubleshooting common issues #

Common issues and their solutions:

  • Compilation Errors: Carefully review compiler error messages. They usually provide helpful information to pinpoint the location and nature of the problem in your SCORE code.
  • Deployment Failures: Double-check your deployment transaction details. Ensure that the contract bytecode and other parameters are correct. Check sufficient balance for transaction fees. Examine the transaction logs for clues to identify the issue.
  • Runtime Errors: Examine the transaction logs for error messages. These can pinpoint runtime errors, such as exceptions or invalid inputs. Use debugging tools to step through the execution flow.
  • Unexpected Behavior: Review your contract’s logic thoroughly. Use debugging tools to trace the execution flow and identify the source of the unexpected behavior. Consider adding more logging statements to help you understand what’s happening.
  • Network Issues: Ensure your development environment is correctly configured to interact with the Icon network. Check network connectivity and the status of the network nodes.

Remember to consult the official Icon documentation and community resources for more in-depth information and solutions to specific problems.
